# Friendly Demo Report Design v0.1

**Status:** Design only — not implemented  
**Repository:** [`ijamhour/SDVM-Action-Preview`](https://github.com/ijamhour/SDVM-Action-Preview) (public)  
**Scope lock:** Defines a presentation-oriented demo artifact. Does **not** implement the demo, publish Marketplace, publish the draft release, or add private engine code.

## Purpose

Design a **public-safe friendly demo report** that helps non-technical and semi-technical readers understand the value of the SDVM Action Preview without implying that this public repository contains the full SDVM engine.

The friendly demo is a **static, synthetic, presentation-oriented illustration** of how public-safe evidence screening can be communicated. It is **not** the full SDVM diagnostic report and is **not** produced by the private SDVM engine.

## Audience

| Audience | What they should understand |
|----------|----------------------------|
| **Technical reviewer** | The preview has a real executable evidence-screening surface — JSONL validation, sufficiency heuristics, PRE/POST/DELTA shape detection, and Markdown/JSON preview reports via CLI and GitHub Action. |
| **Potential evaluator / workflow owner** | The preview helps decide whether a workflow has enough evidence shape for a controlled comparison — not whether SDVM has already diagnosed the workflow. |
| **Executive / non-technical reader** | SDVM is about **operational evidence** and **workflow coordination** readiness, not generic AI performance claims or black-box scores. |

## Demo role in the SDVM story

The public repository now has two complementary layers:

| Layer | Role |
|-------|------|
| **1. Action Preview** | Executable technical layer — validates evidence structure, runs screening heuristics, emits Markdown/JSON preview reports. |
| **2. Friendly Demo Report** | Presentation layer — human-readable interpretation of what screening *could* look like for stakeholders who will not read raw JSONL or CLI output. |

**Action Preview answers:**

> Do we have evidence shape sufficient for screening?

**Full private SDVM answers:**

> What does the evidence reveal about workflow coordination, degradation, and intervention?

The friendly demo report bridges README/CLI output and pilot intake conversations. It does **not** replace Action Preview outputs or private SDVM analysis.

## What the demo should show

- A **synthetic** recurring AI-assisted workflow with multiple runs.
- Canonical JSONL fields in plain language: `run_id`, `ts`, `actor`, `event_type`.
- How Action Preview classifies evidence sufficiency (`sufficient` / `partial` / `insufficient`).
- Whether PRE/POST/DELTA structure is present, partial, or absent — without delta scoring.
- A clear **candidate next step** (e.g., prepare authorized pilot trace package).
- Explicit **limitations** and a path back to README, Action usage, and pilot intake.

## What the demo must not imply

- Full SDVM engine or pilot-grade analysis.
- Measured-variable scoring, S/D/V/M diagnostic scoring, or SDVM scores.
- Statistical methodology, thresholds, weights, normalization, or calibration.
- Playbook recommendations or intervention prescriptions.
- Commercial product UI, hosted SaaS, or Marketplace publication.
- Gate 3 completion or commercial-ready status.
- That the friendly demo is generated by the private SDVM runtime.

## Recommended demo flow

1. **Start with the README** — this is a public-safe preview slice, not the full product.
2. **Show synthetic evidence** — explain `run_id`, `ts`, `actor`, `event_type` in a recurring workflow context.
3. **Show Action/CLI output** — point to actual Markdown/JSON preview report artifacts (`validation.json`, `preview_report.json`, `preview_report.md`).
4. **Show friendly demo report** — human-readable interpretation of the screening result for sponsors and executives.
5. **Close with controlled pilot** — full SDVM analysis remains in the private repository and authorized pilot work only.

## Proposed report sections

The friendly demo report should include these sections in order:

1. **Header:** SDVM Action Preview — Friendly Sample Report
2. **Subtitle:** Synthetic illustration of public-safe evidence screening
3. **Preview notice** — static, synthetic, not pilot-grade analysis
4. **Evidence summary** — run count, actors, event types, time span (synthetic)
5. **Sufficiency status** — screening classification in plain language
6. **PRE/POST/DELTA shape** — present / partial / absent (no delta scoring)
7. **What this suggests** — categorical interpretation only
8. **What remains unknown** — explicit gaps (no scoring, no causal claims)
9. **Candidate next step** — e.g., prepare authorized pilot trace package
10. **Limitations** — public-safe scope restatement
11. **Links** — README, Action usage, pilot intake (`https://sdvm.tech/pilot/intake/`)

### Allowed section labels

- Evidence structure
- Screening status
- PRE/POST/DELTA shape
- Candidate next step
- Limitations

### Labels to avoid (imply full engine)

- SDVM score
- S/D/V/M score
- diagnosis
- calibration
- playbook recommendation
- intervention prescription
- statistical confidence

## Recommended sample content

**Scenario (synthetic):**

A recurring AI-assisted workflow has multiple runs. Evidence contains actor events, timestamps, event types, and partial PRE/POST structure. The preview report classifies the evidence as **partial but usable for evidence follow-up**, not as a full diagnosis.

**Recommended sample statuses:**

| Field | Sample value |
|-------|--------------|
| Screening status | Partial but usable for evidence follow-up |
| PRE/POST/DELTA shape | Present but incomplete |
| Candidate next step | Prepare authorized pilot trace package |

**Sample “What this suggests” (categorical only):**

- Evidence structure is recognizable and repeatable across runs.
- PRE/POST comparison may be possible after capture refinement.
- Workflow may be a candidate for controlled pilot screening — not for autonomous remediation.

**Sample “What remains unknown”:**

- Workflow coordination degradation patterns (requires full SDVM analysis).
- Causal root cause or intervention effectiveness.
- Any scored diagnostic or playbook output.

## Copy guardrails

Include this wording (or equivalent) prominently in every friendly demo artifact:

> This friendly report is **synthetic** and **presentation-oriented**. It does **not** represent full SDVM pilot-grade analysis. It does **not** include scoring internals, statistical methodology, thresholds, weights, calibration, or playbook logic.

Additional guardrails:

- Always pair “screening” with “preview” or “public-safe” in titles visible to external readers.
- Never use “diagnosis,” “score,” or “confidence interval” in friendly demo copy.
- Distinguish **Action Preview outputs** (executable, Markdown/JSON) from **friendly demo** (static illustration).
- Link to [`SECURITY.md`](../../SECURITY.md) posture: no secrets, no customer traces in public issues.

## Relationship to Action Preview outputs

| Action Preview (executable) | Friendly demo (static) |
|----------------------------|------------------------|
| Produced by CLI / GitHub Action | Hand-authored or templated from synthetic fixture |
| `validation.json`, `preview_report.json`, `preview_report.md` | `friendly_sample_report.md` (+ optional HTML/JSON) |
| Sufficiency heuristics from real or synthetic JSONL input | Illustrates how to *communicate* screening results |
| Non-blocking; errors fail the step | Read-only; no runner execution required |

The friendly demo should **align semantically** with Action Preview field names (`sufficiency`, PRE/POST/DELTA shape) but may add narrative prose Action Preview reports deliberately omit.

## Relationship to full SDVM

| Surface | Location | Scope |
|---------|----------|-------|
| Action Preview | This public repo | Evidence shape validation, limited screening, Markdown/JSON reports |
| Friendly demo | This public repo (`docs/demo/`) | Presentation layer for selective sharing |
| Full SDVM engine | Private `ijamhour/SDVM` | Coordination diagnostics, scoring, playbooks, calibration — pilot work only |

Readers who need workflow coordination analysis, degradation diagnosis, or intervention guidance must go through **controlled pilot intake** — not infer it from the friendly demo.

## Risks

| Risk | Mitigation |
|------|------------|
| Friendly demo mistaken for live engine output | Prominent synthetic/preview notice; separate path from CLI artifacts |
| HTML demo implies product UI | Label as static sample; no interactive dashboard claims |
| Sponsor over-interprets “partial but usable” as approval to deploy | “Candidate next step” points to authorized pilot package, not production go-live |
| Technical reviewer dismisses demo as “just marketing” | Link demo sections to real CLI output and `examples/synthetic/` fixtures |
| Copy drift toward scoring language | Allowed/forbidden label lists; copy guardrails block |

## Recommendation

**Proceed to static Markdown + HTML friendly demo report**, provided it remains clearly synthetic and public-safe.

Implementation sequence:

1. **First PR:** `friendly_sample_report.md` + `docs/demo/README.md` (Markdown-only acceptable minimum).
2. **Follow-up PR (optional):** static `friendly_sample_report.html` and `friendly_sample_report.json` derived from the same synthetic scenario.
3. **Optional:** screenshot asset for README selective-sharing section — synthetic only, no stale full-engine framing.

Do **not** wire friendly demo generation into CLI or Action runtime in the public repo without a separate design review.

## Non-goals

- Implement the friendly demo in this design PR
- Publish GitHub Marketplace listing
- Publish draft release `sdvm-action-preview-public-v0.1`
- Create or move tags
- Add private engine code or imports from `sdvm/`, `sdvm_playbooks/`, `sdvm_baseline/`
- Expose scoring internals, statistical methodology, thresholds, weights, normalization, calibration, or playbook logic
- Imply Gate 3 complete or commercial-ready status
- Generate HTML reports from Action Preview runtime (friendly demo is static illustration only)
